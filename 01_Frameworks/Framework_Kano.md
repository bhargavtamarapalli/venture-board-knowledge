# Kano Model Framework

**Version:** 1.0

## Purpose

Evaluate and prioritize product features based on their impact on customer satisfaction rather than implementation effort alone.

## When To Use

- MVP definition
- Feature prioritization
- Product roadmap planning
- Customer research
- Product-market fit evaluation

## Inputs Required

- Customer interviews
- Feature backlog
- Product vision
- Competitive analysis

## Core Concepts

The Kano Model classifies features into five categories:
- Must-Be
- Performance
- Attractive
- Indifferent
- Reverse

## Framework

### 1. Must-Be
Basic expectations. Absence causes dissatisfaction.

### 2. Performance
Better performance increases satisfaction proportionally.

### 3. Attractive
Unexpected features that delight customers.

### 4. Indifferent
Features customers do not value.

### 5. Reverse
Features that reduce satisfaction for some users.

## Evaluation Checklist

- Is this feature expected?
- Does it differentiate the product?
- Is there evidence from customers?
- Is it required for MVP?

## Questions To Ask

- What happens if the feature is absent?
- Will customers pay for it?
- Does it improve retention?
- Does it create competitive advantage?

## Metrics

- Satisfaction
- Adoption
- Usage
- Retention
- NPS

## Red Flags

- Treating every feature as Must-Have
- No customer validation
- Feature creep

## Best Practices

- Validate with customer interviews.
- Reclassify after research.
- Combine with MoSCoW and RICE.

## Common Mistakes

- Prioritizing opinions over evidence.
- Ignoring customer expectations.

## Output Format

Produce:
- Kano Classification Table
- Priority Recommendations
- MVP Guidance
- Risks
